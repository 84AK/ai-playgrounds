# 사용자 이슈 원인 파악 및 수리 작업 로그 (2026-04-09)

## 1. 30명 동시 회원가입/로그인 시 "아이디 찾을 수 없음" 현상
**원인: 구글 앱스 스크립트의 동시성 잠금장치(Lock) 타임아웃**
* 현재 서버(구글 앱스 스크립트) 코드에는 여러 명이 동시에 데이터를 쓸 때 꼬이지 않도록 `lock.waitLock(20000);` 이라는 자물쇠 기능이 있습니다. (최대 20초 대기)
* 한 반 30명이 "동시에 땡!" 하고 가입하기를 누르면 줄을 서게 되는데, 20초 안에 자신의 차례가 오지 않은 학생들의 요청은 앱스 스크립트가 강제로 거절(Timeout)하게 만들어 버립니다.
* 결국 이 학생들은 화면 상단에 "서버 렉/결과 미확인" 에러 스낵바가 떴겠지만 무시하고 다시 로그인을 시도했을 것입니다. 하지만 실제로 구글 시트(DB)에는 기록되지 못했기 때문에 "없는 아이디"라고 뜨게 됩니다.

**🛠️ 해결 조치 사항:**
1. **클라이언트 재시도 (Exponential Backoff + Jitter) 로직 탑재 완료**: `lib/appsScriptClient.ts`의 `postAppsScript` 함수에서 `500 Server Error`나 타임아웃이 발생하면 포기하지 않고 지수형 대기(랜덤 시간 지연) 방식으로 **최대 3회** 자체적으로 무음 재시도하도록 자동화했습니다.
   - 30명이 눌러도 맨 뒤의 10명이 타임아웃이 나면 알아서 몇 초 뒤 다시 줄을 섭니다! (학생은 버튼 누르고 10초 정도 가만히 있으면 결국 가입 성공 화면으로 진입함)
2. **백엔드(Apps Script) 잠금 시간 최대 상향**: `Docs/updated_gas_code.js` 내부 서버 코드에서 `lock.waitLock(20000)`을 구글의 최대 허용치인 `lock.waitLock(30000)` (30초)로 10초 늘렸습니다!

## 2. 프로필 수정했음에도 과제가 '기타' 폴더로 저장되는 기이한 현상
**원인: 기기 간/브라우저 캐시(Local Storage) 동기화 누락**
선생님 컴퓨터에서 직접 그 학생 아이디로 "로그인" 해보니 잘 되었다고 하셨죠? 이 사실이 가장 명확한 단서를 제공합니다! 구글 스프레드시트 서버에는 학생의 학년/반 데이터가 **확실하게 정상적으로 저장되어 있습니다.** 

하지만 학생 컴퓨터에서 기타 폴더로 간 이유는 아래와 같습니다.
1. 학생이 A 기기(패드, 폰 등) 또는 선생님이 구글 시트에서 학년/반을 세팅합니다.
2. 학생이 과제를 제출하러 B 기기(노트북, 집 컴퓨터 등)를 엽니다. 이 컴퓨터에는 "예전에 학년/반 정보가 생기기 전" 로그인했던 정보가 구글 크롬에 남아있습니다.
3. 우리 웹앱의 `GlobalAuthGuard.tsx`가 영리하게 "자동 로그인"을 시켜주는 과정에서 큰 실수가 하나 있습니다! **비밀번호가 맞으면 무조건 바로 통과시켜주는데, 이때 최신 정보(학년/반 등)를 구글 시트로부터 가져와서 자신의 기기에 업데이트해 주지 않습니다.**
4. 결국 노트북은 학년/반을 모르는 상태(과거 상태)이고, 학생은 과제를 올립니다. 시스템은 "학년/반 정보가 없네? '기타' 폴더로 가라" 라고 판단합니다.
5. 선생님이 선생님 컴퓨터로 그 아이디로 로그인할 때는 **스프레드시트에 있는 "최신정보"를 싹다 가져와서 로컬에 새롭게 저장하며 로그인(초기 1회 갱신)**하기 때문에 아무런 문제 없이 작동한 것입니다!

**🛠️ 해결 조치 사항:**
* **자동로그인 정보 갱신(캐시 동기화) 적용 완료**: `components/GlobalAuthGuard.tsx`에서 학생이 "자동 갱신"되어 통과될 때, 그냥 스쳐지나가지 않고 백그라운드에서 방금 서버가 말해준 **학년, 반, 아바타 등 최신 정보 패키지를 무조건 브라우저 로컬 저장소에 덮어씌워서(`writeLocalProfile`) 저장**하도록 로직을 수정했습니다! 이제 어느 컴퓨터에서 열어도 무조건 서버와 100% 동일한 학년 반 정보를 갖게 됩니다.
