# 2026-04-10 작업 로그 (서기)

## 에러 및 문제 상황 (의도 파악 기록)
사용자가 "드라이브와 구글 스프레드시트에 박선우 학생의 3주차 과제 제출 내역이 명확히 보이는데, 랭킹 페이지에서는 3주차 과제 제출 여부가 체크되어 있지 않다"며 문의를 주었습니다.

## 분석 (상담가/해결자 의견)
해당 이슈의 원인은 다음과 같이 분석되었습니다.
1. **백엔드 매핑 기준:** 구글 앱스 스크립트 기반 백엔드(`Docs/Code_v5_2_with_Ranking.js`)에서는 1주차~4주차의 MBTI 진도 데이터를 크기가 4인 배열로 묶어서 반환합니다. (즉, 인덱스 0이 1주차, 인덱스 1이 2주차, 인덱스 2가 3주차, 인덱스 3이 4주차 데이터)
2. **프론트엔드 라벨링 오류:** 랭킹 카드 컴포넌트(`components/RankingCard.tsx`)에서 MBTI 진행 상황을 렌더링할 때, 배열의 인덱스 값을 그대로 라벨(`0, 1, 2, 3`)로 사용하고 있었습니다. (POSE 과제는 `1, 2, 3, 4`로 정상 표기됨)
3. **오해 발생 패턴:** 박선우 학생은 3주차(스프레드시트 3번째 컬럼, 배열의 인덱스 2)에 정상적으로 과제를 제출하여 데이터 값이 TRUE 처리되었습니다. 따라서 랭킹 UI에서 `0, 1, 2` 라벨 항목이 체크되었고, `3` 라벨이 미제출 상태로 떴습니다. 사용자는 이 `3` 라벨을 3주차로 받아들여 제출 체크가 누락된 것으로 오해했던 것입니다.

## 해결 방법 및 구현 (작업자)
- **수정 위치:** `components/RankingCard.tsx` 70~80번째 줄 부근
- **수정 내용:** MBTI 진행도를 보여주는 배열 매핑(`map`) 시 렌더링 값(화면에 보여주는 숫자)과 호버 말풍선(title)의 주차 숫자에 `+1`을 더하도록 수정 (`i` -> `i + 1`).
- 이를 통해 MBTI도 POSE와 동일하게 `1, 2, 3, 4주차`로 정상 라벨링됨으로써 직관적인 진행도 파악이 가능해졌습니다. 이제 박선우 학생의 페이지는 `1, 2, 3`이 체크된 정상적인 모습(1~3주차 제출 완료)으로 올바르게 나타납니다.

## 결과 및 후속 조치
- 랭킹 보드의 `MBTI` 주차 표기가 0부터 시작하던 버그가 해결되어 실제 데이터와 UI 간 혼동이 사라졌습니다.
- 추가적인 스프레드시트 구조 변경 없이 직관적으로 사용자 경험이 개선되었습니다.
