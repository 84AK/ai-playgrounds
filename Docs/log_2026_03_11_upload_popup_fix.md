# 작업 로그: 2026-03-11 파일 업로드 팝업 오류 수정 및 검증 완수

## 1. 개요
* **날짜:** 2026-03-11
* **작업자:** 서기(Doc) / 해결사(Fix)
* **목적:** 구글 드라이브 파일 업로드 기능 정상화 및 프론트엔드 검증 로직 매칭 수정

## 2. 발생했던 문제 (Issue)
1. 사용자가 1주차 과제 제출(Google Drive 업로드)을 진행했을 때, 드라이브 상에는 정상적으로 압축 파일이 업로드되었으나, 화면에서는 **"업로드 실패"** 팝업이 노출되는 심각한 UX 문제 발생.

## 3. 원인 파악 (Root Cause)
* **백엔드 (Apps Script V3):** 구글 드라이브 파일 업로드 후 Progress 시트의 성취도를 업데이트할 때 헤더의 컬럼 이름을 `MBTI_Week1`, `POSE_Week1`과 같은 형태로 기록하고 있었음.
* **프론트엔드 (`UploadHomework.tsx`):** 업로드 완료 직후 앱스 스크립트의 `getProgress` API를 호출해 결과를 검증할 때, 반환된 응답 데이터에서 과거 버전의 포맷인 `week1` 키값을 기대함.
* **결론:** 실제 드라이브 업로드 및 시트 업데이트는 **완벽하게 성공**했으나, 프론트엔드가 응답 키 불일치(`week1` vs `mbti_week1`)로 인해 무조건 업로드 실패 처리 및 팝업을 띄우는 논리적 오류 확인.

## 4. 해결 로직 (Resolution)
1. **`app/course/[weekId]/UploadHomework.tsx` 수정:**
   - 기존의 엄격한 `week1` 키 검증 방식을 개선.
   - 앱스 스크립트 V3가 반환하는 새로운 데이터 구조에 맞추어 `mbti_week1` 형식의 키값과 기존 `week1` 형식의 키값을 모두 확인할 수 있도록 로직 유연화 (`isSuccess` 변수 도입).
   - 변경된 코드: `const isSuccess = checkResult?.data?.[`mbti_week${weekId}`] === true || checkResult?.data?.[`week${weekId}`] === true;`
   - 이로 인해 실제 업로드 성공 시에도 정상적으로 **"성공"** 팝업 노출.

## 5. 결과 및 기대 효과
* 앱스 스크립트와 프론트엔드 간의 데이터 동기화 완료.
* 사용자들에게 올바른 업로드 완료 메시지를 보여줌으로써 혼선 방지 및 안심 제공.
* 이후 주차(2, 3, 4주차) 및 다른 코스(POSE 등)에서도 동일하게 유연한 검증이 가능하도록 토대 마련.

## 6. 다음 진행 예정 사항
* 향후 코스 메뉴(네비게이션 바 등)에서의 실시간 진도율 업데이트 연동이 필요한 경우, `getProgress` 응답 형태(`mbti_week1` 등)를 기준으로 통일성 고려 필요.
